\section{Overview}

This chapter presents an overview of the design, implementation, and validation of an edge-based Battery Management System (BMS). The objective of this research is to develop a decentralized and intelligent BMS architecture capable of real-time data processing and accurate state estimation.

The methodological framework follows a data-driven modeling, training, and deployment paradigm, motivated by the growing demand for localized computation in next-generation BMSs. Traditional cloud-centered architectures often suffer from latency, bandwidth limits, and vulnerability to network interruptions. By integrating data processing, model training, and experimental deployment, this research aims to achieve a low-latency and updatable edge BMS.

The general research process consists of five main stages.

1. \textbf{Data Acquisition and Preprocessing}

Battery operational data, including voltage, current, temperature, relative time, and state-of-charge (SOC) readings, are collected from experimental setups and public datasets. The raw data are preprocessed by noise filtering, normalization, segmentation, and feature extraction to ensure data quality for both modeling and embedded inference.

2. \textbf{Modeling and Algorithm Development}

This stage focuses on developing data-driven SOC estimation models. LightGBM decision trees are trained on historical data and optimized through systematic hyperparameter search. The models are evaluated on validation and test datasets to ensure accuracy and robustness under different load and temperature conditions.

3. \textbf{Model Compilation and Deployment}

The trained model is converted into C code using Efficient Decision tree Ensembles (Eden) \autocite{Eden} for embedded execution. Both the model and feature extraction logic are adapted for compatibility with the target hardware. The hardware development board and software environment are configured using SPC5Studio, which integrates the generated code and enables onboard preprocessing and real-time SOC inference.

4. \textbf{Performance Verification}

The deployed inference program is evaluated on the embedded board with representative datasets. The verification process includes functional validation and performance analysis. Memory usage and CPU cycle count are measured through custom scripts to evaluate computational efficiency and real-time capability.

5. \textbf{Over-the-Air (OTA) Update Mechanism}

An OTA mechanism is implemented to support remote model updates. This allows new or improved SOC estimation models to be securely deployed without manual intervention. The OTA system ensures long-term adaptability and maintains high performance in field operation.

To better illustrate how the designed architecture operates as an integrated system, the operational logic of the deployed framework is presented in \autoref{fig:project_architecture}. This figure demonstrates the interaction between onboard estimation and cloud-based model management.

(1) Battery data are collected by sensors and stored in the onboard memory.

(2) Onboard SoH and SoC models perform real-time estimation.

(3) Data are periodically uploaded to the cloud for analysis.

(4) When retraining conditions are met, the cloud updates the SoC model using new data.

(5) The updated model is transmitted back to the vehicle via OTA, ensuring continuous improvement of estimation accuracy.

% 2. The data collected are used as input for the SoH and SoC on-board estimation models. These models operate in real time to monitor battery status during operation.

% 3. The stored data are periodically uploaded to the cloud for further processing. This upload is triggered when certain conditions are met, such as reaching a predefined time interval or when the onboard memory approaches its storage limit. 

% 4. When additional retraining conditions are satisfied, such as a drop in SoH 1\% or the elapsed update period, the cloud retrains the SoC estimation model using the latest accumulated battery data.

% 5. The updated SoC model is transmitted from the cloud back to the edge node of the EV. The edge node replaces the previous model with the new version, and the system resumes from Step 2 for continued operation.

In this step, the input signals for each discharge cycle are divided into fixed-length sliding windows without overlap. We use window lengths $wlen \in \{20, 30, 40, 50, 60\}$, which are commonly used in the literature. Users can adjust the length or apply sliding windows with overlapping based on their needs or the sampling frequency of the data set.
